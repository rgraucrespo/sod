General philosophy for input generation
-------------------------------------
SOD enumerates inequivalent supercell configurations and writes calculator‐specific input files for each configuration.  To make the workflow uniform and extensible across calculators, a single template file is used per calculator and SOD only inserts structural information and configuration numbers.  **Template names follow the pattern `template_<real_input_name>`** (e.g. `template_input.gin` for GULP).  The user edits a template that looks like a normal input file for their calculator, marks where the structure goes with a token, and SOD creates one folder per configuration (`cXX`) inside a composition folder (`nXX`).  SOD writes the completed input file (and any auxiliary data file, if needed) in each `cXX` folder.  All calculator‐specific run options, cut‑offs, species definitions and k‑points remain in the user template; SOD only inserts the structure and configuration number.

The folder structure is unified: for each composition (defined by the number of substitutions `XX`) SOD creates a folder `nXX/`, and inside it one folder `cYY/` for each configuration `YY`.  This organisation isolates outputs and makes run scripts simple.  Configuration numbers are zero‑padded based on the total number of configurations.  Each calculator section below refers to this structure and uses the template naming convention.

GULP – `template_input.gin`
---------------------------
1. **Template**: The user provides one file `template_input.gin` that looks like a normal GULP input.  It must contain a line on its own with `@configuration_structure@` where SOD should insert the generated GULP structure block.  The optional token `@configuration_number@` can be used in titles or filenames.  All other GULP input (keywords, optimisation flags, species/potential definitions, output directives) remains unchanged.
2. **Library handling**: If the template contains a `library name.lib` directive, SOD reads `name.lib` from the main directory, extracts only the entries needed for the GULP atom types used in the problem, and writes a reduced `name.lib` into each `cXX` folder.  This reduced library preserves all information needed to run GULP (charges, shell definitions, potentials, springs, etc.) for the relevant types.  Even if a library of the same name exists in GULP’s standard search path, SOD still copies the reduced library so that each configuration folder is self‑contained.  If potentials and charges are given directly in `template_input.gin` (inline definitions), they will be inlined into each generated input and no library file is copied.  SOD never inlines the contents of an external library.
3. **Species mapping**: SOD species names and GULP atom types are not always the same.  If the SOD species label differs from the force‑field atom type, the user must add comment lines in the template of the form `# sod_type_map <SOD_species> <GULP_type>`.  Distinct GULP atom types should correspond to distinct SOD species.  In particular, if two chemically identical species must use different GULP types (for example `O1` vs `O2`), they should be defined as separate SOD species; if the SOD names differ from the type names, mapping lines must be given.
4. **Core–shell inference**: The GULP shell model is determined from the force‑field definitions or from the reduced library, not from `INSOD`.  SOD reads the inline species definitions and the reduced library to decide whether each type is core‑only or core+shell.  Core/shell declarations and flags like `genarc`/`genxtl` should be removed from `INSOD`.
5. **Output and naming**: Because each configuration has its own directory `cYY`, output filenames inside the GULP template need not include configuration numbers.  Titles or labels may use `@configuration_number@` if desired.  SOD writes `input.gin` and the reduced library (if any) into each `cYY` folder.

A typical template_input.gin file could look like:

     opti conp
     title
       Spinel configuration @configuration_number@
     end

     @configuration_structure@

     library catlow.lib

     output arc spinel.arc
     output xtl spinel.xtl


LAMMPS – `template_conf.data`
-----------------------------
1. **Template**: The user provides a single file `template_conf.data`, which should look like a normal LAMMPS data file containing counts, box bounds, masses and coefficients, followed by the structure.  The template must include tokens that SOD will replace when generating a configuration.  These tokens include `@num_atoms@`, `@num_bonds@`, `@num_atom_types@`, `@num_bond_types@`, `@box_bounds@` and `@configuration_structure@`.  The optional token `@configuration_number@` may be used in comments or file names.  All other text (including the `Masses` and `Pair Coeffs` sections, if present) remains unchanged.  The `@configuration_structure@` token should be placed at the start of the `Atoms` (and `Bonds` if needed) section to mark where SOD should insert the atom list with IDs, molecule IDs, types, charges and coordinates (and the core–shell bonds if needed).
2. **Script generation**: SOD provides a simple, static `in.lammps` script in each configuration folder to run the simulation.  This script typically contains commands such as `units`, `atom_style`, `read_data conf.data`, `pair_style`, `bond_style`, `pair_coeff`, `bond_coeff`, and any other run controls.  Users may edit this script outside of SOD if needed, but they do not provide a template for it.  Instead, the focus of templating is on the data file containing the structure.
3. **Species mapping and roles**: Distinct LAMMPS atom types must correspond to distinct SOD species.  Mapping lines should appear as comments in `template_conf.data` of the form `# sod_type_map <SOD_species> <role> type=<N> [bond_type=<M>]`, where `<role>` is `core` or `shell`.  These lines tell SOD which numeric type to assign to each species and, if the species is represented as a core–shell pair, which bond type to use for the core–shell spring.  SOD uses these mappings when filling the `@configuration_structure@` section.  If no mapping is provided for a species, SOD assumes that the SOD name matches a LAMMPS type only when this is obvious; otherwise it raises an error.
4. **Data file tokens**: `template_conf.data` may include additional tokens for convenience.  For example `@masses@` can mark a section where SOD writes the `Masses` for each atom type; `@pair_coeffs@` can mark a section for `Pair Coeffs` lines.  If the user includes these tokens, SOD will replace them with the appropriate lines derived from the species mapping and from the template itself.  If these sections are omitted, the corresponding information should appear in the static `in.lammps` script.
5. **Folder contents**: Each `cYY` folder contains two files: a static `in.lammps` script and the generated `conf.data` file based on `template_conf.data`.  Additional outputs generated by LAMMPS (logs, dumps, restart files) are controlled in the static script.

Example of template_conf.data:

LAMMPS data file for SOD configuration @configuration_number@

@num_atoms@ atoms
@num_bonds@ bonds

@num_atom_types@ atom types
@num_bond_types@ bond types

@box_bounds@

Masses

1 65.38
2 55.845
3 15.9994
4 0.0

# sod_type_map Zn core  type=1
# sod_type_map Fe core  type=2
# sod_type_map O1 core  type=3
# sod_type_map O1 shell type=4 bond_type=1

Atoms # full

@configuration_structure@


VASP – `POSCAR`
---------------
1. **Purely structural input**: VASP separates the structure (`POSCAR`) from run settings (`INCAR`), k‑point sampling (`KPOINTS`) and pseudopotentials (`POTCAR`).  Therefore SOD does not need a template file for VASP.  It generates only the `POSCAR` file for each configuration.
2. **Folder structure**: SOD creates `POSCAR` inside each `cYY` folder under `nXX/`.  The user must supply the accompanying `INCAR`, `KPOINTS` and `POTCAR` files.
3. **Species ordering**: VASP’s `POSCAR` lists species in a particular order that must match the order of potentials in `POTCAR`.  SOD chooses a deterministic ordering (for example alphabetical or as given in `INSOD`) and uses it consistently across all configurations.  The user must ensure that `POTCAR` follows the same order.

Quantum ESPRESSO – `template_pw.in`
------------------------------------
1. **Template**: The user provides `template_pw.in`, which should look like a normal QE `pw.x` input.  It must contain a `@configuration_structure@` token on a line by itself.  All namelists (`&CONTROL`, `&SYSTEM`, `&ELECTRONS`, etc.), `ATOMIC_SPECIES`, `K_POINTS` and run controls remain in the template.
2. **Structure insertion**: SOD assumes `ibrav = 0` so that it can provide explicit cell parameters.  For each configuration, it replaces `@configuration_structure@` with a block containing `CELL_PARAMETERS` and `ATOMIC_POSITIONS`.  The user defines `ATOMIC_SPECIES` explicitly in the template; SOD does not modify this card.
3. **Configuration numbers**: The optional token `@configuration_number@` may be used in titles, prefixes or output filenames.
4. **Pseudopotentials**: QE pseudopotential filenames are specified in `ATOMIC_SPECIES` and found through the path in `pseudo_dir` (in `&CONTROL`).  SOD does not copy pseudopotential files; it leaves `pseudo_dir` under user control.  The user must ensure that `pseudo_dir` is valid from within each configuration folder or uses relative paths.
5. **Folder contents**: For each configuration, SOD writes `pw.in` into `cYY/`.  Because the template includes all run settings, SOD does not modify any other part of the input.

A typical template_pw.in file could look like:

     &CONTROL
       calculation = 'scf'
       prefix = 'conf_@configuration_number@'
       outdir = './tmp'
       pseudo_dir = '../../pseudo'
     /
     &SYSTEM
       ibrav = 0
       nat = 32
       ntyp = 3
       ecutwfc = 60.0
       ecutrho = 480.0
       occupations = 'smearing'
       smearing = 'mv'
       degauss = 0.02
     /
     &ELECTRONS
       conv_thr = 1.0d-8
     /

     ATOMIC_SPECIES
     Zn  65.38   Zn.upf
     Fe  55.845  Fe.upf
     O   15.999  O.upf

     @configuration_structure@

     K_POINTS automatic
     4 4 4 0 0 0



CASTEP – `template_castep.cell`
-------------------------------
1. **Template**: The user provides a single file `template_castep.cell` that looks like a real CASTEP `.cell` file and includes a single `@configuration_structure@` token on a line by itself.  All fixed `.cell` content (such as symmetry, k‑point sampling, constraints, external pressure, etc.) is written by the user in this template.
2. **Structure insertion**: For each configuration, SOD replaces `@configuration_structure@` with the configuration‑specific lattice and positions, typically as `%BLOCK LATTICE_CART` (or `LATTICE_ABC`) and `%BLOCK POSITIONS_FRAC` (or `POSITIONS_ABS`).  The user controls which block style to use by how they write the template.
3. **Parameter file**: CASTEP separates structure (`.cell`) from calculation parameters (`.param`).  SOD does not manage the `.param` file.  The user should place an appropriate `.param` (e.g. `castep.param`) in each `cYY/` folder or reference it from the run script.  The seed name must match; if the generated structural file is `castep.cell`, the `.param` file must be named `castep.param`.
4. **Folder contents**: Each `cYY` folder will contain `castep.cell` generated by SOD.  Additional files (`castep.param`, basis set files, pseudopotentials) remain under user control.

Notes on numbering and folders
-----------------------------
- **Zero‑padding**: SOD uses flexible zero‑padding based on the total number of configurations.  If there are 200 configurations, configuration numbers run from `001` to `200`.  If there are 15, they run from `01` to `15`.  Composition folders `nXX` are similarly zero‑padded.
- **Self‑contained directories**: Copying reduced libraries (for GULP) and isolating outputs in `cYY` folders ensures each configuration folder is self‑contained.  Users may delete unwanted copies (e.g. library files) after generation if they wish.
- **User responsibility**: SOD does not manage pseudopotentials (QE), `.param` files (CASTEP), `INCAR/KPOINTS/POTCAR` (VASP) or LAMMPS outputs beyond the data file.  The user must supply these as needed.

This unified approach preserves the native look of calculator inputs while removing calculator‑specific options from `INSOD`.  By using a single template per calculator and consistent folder names, SOD simplifies both input preparation and post‑processing across GULP, LAMMPS, VASP, CASTEP and Quantum ESPRESSO.
